大家好，我是「山丘代码铺」。

做 RAG、搜索、知识库的时候，经常会看到一个词：

> **BM25。**

它听起来像某种复杂模型。

但其实不是。

BM25 不是大模型，也不是向量检索，而是一套经典的关键词相关性打分算法。

它要解决的问题很简单：

> **当用户输入一个查询时，哪些文档应该排在前面？**

你给它一个查询，比如：

```text
Redis Stream 消息队列
```

它会去看每篇文档里：

```text
Redis 出现了吗？
Stream 出现了吗？
消息队列 出现了吗？
这些词出现得多不多？
这些词在全库里常不常见？
这篇文档是不是特别长？
```

然后给每篇文档算一个相关性分数。

分数越高，越排前面。

很多 RAG 系统效果不好，不一定是大模型不行。

也可能是第一步资料没找对。

BM25 的价值，正是在这里。

所以这篇先记住一句话：

> **BM25 不是理解你在说什么，而是在判断哪篇文档更像你要找的那篇。**

---

## 01｜为什么不能只数关键词？

先从一个最朴素的问题开始。

假设你在知识库里搜：

```text
登录失败 token 过期
```

系统里有两篇文档。

第一篇很短：

```text
登录失败时，如果 token 已过期，需要重新获取 token。
```

第二篇很长：

```text
这里是一份完整的认证系统设计文档。
里面讲了登录、注册、短信验证码、token、refresh token、权限校验、
网关、风控、日志、审计、异常处理……
```

如果只是简单数关键词，第二篇可能赢。

因为它很长，里面出现了很多相关词。

但你真正想看的，可能是第一篇。

因为第一篇虽然短，但它几乎正好回答了你的问题。

这就是搜索排序里的第一个坑：

> **不能谁词多就谁排前面。**

长文档天然更容易命中关键词。

如果不处理，很多大而全的文档会压过真正精准的小文档。

BM25 就是在解决这类问题。

它不是简单数词。

它会考虑：

```text
这个词在当前文档里出现了多少次
这个词在整个文档集合里有多稀有
这篇文档是不是比平均文档长很多
```

这三件事，基本就是理解 BM25 的入口。

所以，搜索排序不是简单回答“出现了几次”。

而是要判断：

> **这些词出现得有没有价值。**

---

## 02｜BM25 的第一件事：词出现了几次

搜索一个词，如果某篇文档里完全没出现，那相关性通常就低。

如果出现了一次，说明有点关系。

如果出现了好几次，说明关系可能更强。

这个很好理解。

比如你搜：

```text
webhook 签名
```

一篇文档里只在某个角落提了一次 webhook。

另一篇文档反复讲：

```text
webhook 回调
webhook 签名
签名校验
签名失败
```

第二篇大概率更相关。

这就是词频。

英文里常叫：

```text
TF：Term Frequency
```

也就是某个词在某篇文档里出现的次数。

但 BM25 对词频有一个很重要的处理：

> **词出现越多，分数会提高，但不会无限提高。**

这点很关键。

如果一篇文档把“webhook”这个词写了 100 次，它不应该比写了 10 次的文档强 10 倍。

因为词出现到一定程度以后，我们已经知道它相关了。

再重复很多遍，信息增量没那么大。

所以 BM25 会让词频收益逐渐变小。

可以理解成：

```text
出现 0 次：基本没关系
出现 1 次：有明显提升
出现 3 次：再提升一些
出现 10 次：已经很相关
出现 100 次：不会再线性暴涨
```

这叫词频饱和。

它让搜索结果不容易被“关键词堆砌”带偏。

---

## 03｜BM25 的第二件事：这个词有多稀有

不是所有词都一样重要。

比如你搜：

```text
Redis Stream 消息队列
```

这里面有些词信息量更大。

“Redis”很重要。

“Stream”也很重要。

“消息队列”也重要。

但如果查询里有一些很常见的词，比如：

```text
怎么
一个
的
是什么
```

这些词就没有太大区分度。

因为几乎所有文档里都可能出现。

搜索系统真正看重的是：

> **越稀有的词，越能区分文档。**

比如一个技术库里，“系统”“问题”“使用”这种词很常见。

但“BM25”“HMAC”“Redis Stream”“refresh token”这种词更少见。

一篇文档如果命中了这些稀有词，说明它更可能和查询相关。

这就是 IDF。

英文里叫：

```text
IDF：Inverse Document Frequency
```

可以粗略理解成：

> **一个词出现在越少的文档里，它越重要。**

如果某个词几乎每篇文档都有，那它区分不了什么。

如果某个词只在少数文档里出现，那它一出现，就很有价值。

所以 BM25 不只是看“词出现了没有”。

它还看：

> **这个词在整个库里有多难得。**

这也是为什么技术词、产品名、接口名、错误码、类名，通常在搜索里很有用。

它们比“实现”“问题”“使用”这种泛词更能定位文档。

---

## 04｜BM25 的第三件事：文档是不是太长

再回到前面的例子。

长文档有一个天然优势：

> **它字多，所以更容易撞上关键词。**

一篇 300 字的文档，和一篇 30000 字的文档，如果只看命中次数，后者很容易赢。

但这不一定合理。

因为长文档命中关键词，不一定说明它更相关。

它可能只是因为内容太多，什么都沾一点。

所以 BM25 会做文档长度归一化。

简单说就是：

> **同样命中关键词，短文档可能更值得加分；特别长的文档不能因为字多就占便宜。**

比如你搜：

```text
JWT base64url padding
```

一篇短文档专门讲：

```text
JWT 里的 base64url 为什么没有等号
```

另一篇长文档是：

```text
完整认证体系设计，从密码、JWT、OAuth、Session、网关一路讲到审计日志
```

长文档当然也可能有用。

但如果短文档刚好围绕你的查询展开，它应该排得更靠前。

长度归一化就是在处理这个问题。

BM25 会拿当前文档长度和平均文档长度做比较。

文档特别长时，词频加分会被压一压。

文档比较短但命中精准时，分数就更容易冒出来。

这也是 BM25 很实用的地方：

> **它不是只奖励多写，也奖励写得集中。**

---

## 05｜公式不用死背，但可以看一眼

BM25 的公式长这样：

```text
score(D, Q) = Σ IDF(q) ×
              f(q, D) × (k1 + 1)
              --------------------------------
              f(q, D) + k1 × (1 - b + b × |D| / avgdl)
```

第一次看可能有点吓人。

但不用死背。

把它拆成几块就好。

```text
Q：用户查询
D：某篇文档
q：查询里的某个词
f(q, D)：这个词在文档里出现了多少次
|D|：这篇文档的长度
avgdl：所有文档的平均长度
IDF(q)：这个词有多稀有
```

这里的 Σ 表示：

> **对查询里的每个词分别算一遍分数，最后加起来。**

![BM25 的三个打分信号](assets/2026-06-25-bm25-explained-scoring-signals.png)

图：BM25 不是简单数关键词，它会同时考虑词频收益、词的稀有度和文档长度。

再看两个参数：

```text
k1：控制词频收益增长得有多快
b：控制文档长度影响有多大
```

还是那句话，不用一上来钻公式。

公式背后就三件事：

```text
词出现得越多，越相关，但收益会饱和
词越稀有，越重要
文档越长，越不能白占便宜
```

这就是 BM25 的核心。

## 06｜用一个小例子串起来

假设你在一个文章库里搜：

```text
Agent 工具调用
```

有三篇文章。

第一篇：

```text
Agent 如何调用工具？这篇主要讲 Tool Calling 的流程。
```

第二篇：

```text
Agent 是什么？从 ChatBot 讲到 Workflow，再讲到 ReAct 和 Reflection。
```

第三篇：

```text
工具、工具、工具、工具、工具。
这是一个很长的通用工具清单，里面什么工具都提一点。
```

如果只看关键词次数，第三篇可能很强。

因为“工具”出现太多了。

但 BM25 不会让它无限加分。

因为词频会饱和。

再看第一篇。

它同时命中了：

```text
Agent
工具
调用
```

而且文档很集中。

所以它可能排在前面。

第二篇也有 Agent，但没有明显讲“工具调用”。

它可能相关，但没第一篇精准。

这就是 BM25 的直觉：

> **谁更像用户要找的那篇文档，谁排得更靠前。**

不是谁最长。

不是谁堆词最多。

也不是谁看起来最全面。

## 07｜为什么 RAG 里还会用 BM25？

很多人做 RAG 时，会下意识觉得：

> 都用大模型了，还要 BM25 干什么？

其实很要。

因为 RAG 的第一步不是生成答案。

而是先把相关资料找出来。

如果资料找错了，后面模型再聪明，也是在错误上下文里发挥。

BM25 在 RAG 里有几个很实在的价值。

第一，它便宜。

不需要把每段文本都变成向量。

普通倒排索引就能跑。

第二，它可解释。

你能大概看出来：

```text
这篇文档为什么排前面？
因为命中了这些词。
这些词在库里比较稀有。
文档长度也比较合适。
```

第三，它对精确词很稳。

比如查错误码、接口名、配置项、日志关键词，BM25 往往非常直接。

第四，它适合作为召回的一路。

你可以让系统同时跑：

```text
BM25 检索
向量检索
```

再把两边结果合并、去重、重排。

这样既不丢关键词，也能补语义。

很多 RAG 系统最后效果不好，不一定是模型问题。

也可能是检索阶段太单薄。

只靠向量，可能漏掉精确词。

只靠 BM25，可能漏掉语义相近。

所以更稳的思路通常是：

> **先把候选资料召回来，再认真做排序和筛选。**

BM25 就是召回和排序里很重要的一块老地基。

## 08｜BM25 和向量检索有什么区别？

现在一聊搜索，很容易直接想到向量检索。

这两个东西的区别，可以先用两句话记：

> **向量检索擅长“意思差不多”。**
>
> **BM25 擅长“字面上就是它”。**

比如你搜：

```text
接口响应慢
```

向量检索可能找到：

```text
P99 延迟升高
响应时间过长
下游服务超时
```

哪怕这些文档没有完全出现你的原词。

因为它更关心语义相近。

但如果你搜的是：

```text
ERR_INVALID_SIGNATURE
```

或者：

```text
refresh_token_expires_in
```

BM25 往往更稳。

因为你要找的就是这个精确字符串。

它不需要猜意思。

它只要把这个词稳稳命中。

这就是 BM25 的强项。

它擅长找：

```text
产品名
接口名
字段名
错误码
类名
命令
配置项
明确术语
```

向量检索可能理解语义更好，但不一定像 BM25 一样稳稳抓住这些精确词。

所以不要把 BM25 和向量检索理解成谁淘汰谁。

更常见的工程判断是：

> **BM25 负责抓关键词精确命中，向量检索负责补语义相近。**

两个一起用，就是很多人说的混合检索。

![BM25 和向量检索组成混合检索](assets/2026-06-25-bm25-explained-hybrid-retrieval.png)

图：BM25 负责抓关键词精确命中，向量检索补语义相近，最后再合并、去重、重排。

## 09｜BM25 有哪些局限？

BM25 很实用，但它不是万能的。

第一，它不真正理解语义。

你搜：

```text
接口很慢
```

文档里写的是：

```text
响应时间过长
延迟升高
P99 抖动
```

如果词面差太多，BM25 可能找得不够好。

第二，它很依赖分词。

中文搜索尤其明显。

比如：

```text
工具调用
```

到底切成：

```text
工具 / 调用
```

还是：

```text
工具调用
```

会影响命中效果。

技术词、英文缩写、产品名、类名，也都和分词、大小写、符号处理有关。

第三，它不天然知道字段权重。

同一个词出现在标题里，通常比出现在正文角落更重要。

但这需要搜索系统额外设计。

比如给标题、摘要、标签不同权重。

第四，它不替你解决 chunk 切分。

RAG 里如果切块切得太碎，关键词上下文可能被拆散。

如果切得太大，长文档又容易稀释相关性。

所以 BM25 好不好用，不只看算法本身。

还看：

```text
分词怎么做
字段怎么建
文档怎么切
停用词怎么处理
查询怎么改写
结果怎么重排
```

这也是工程里最真实的地方。

一个名字叫 BM25 的算法，只是搜索系统的一部分。

不是把它打开，搜索质量就自动好了。

## 10｜如果面试官问你：什么是 BM25？

如果面试里被问到：

```text
你怎么理解 BM25？
```

不要一上来背公式。

可以先这样答：

> **BM25 是一种经典的关键词相关性排序算法，常用于全文检索。它会根据查询词在文档中的出现次数、查询词在整个文档集合中的稀有程度，以及文档长度，对文档进行打分排序。**

然后补上三件核心事：

```text
词频：词在文档里出现得越多，相关性通常越高，但收益会饱和。
IDF：词越稀有，越能区分文档，权重越高。
长度归一化：长文档不能因为字多就天然占便宜。
```

如果要和向量检索对比，可以说：

> **BM25 更偏关键词精确匹配，对错误码、接口名、字段名、产品名这类词很稳；向量检索更偏语义相似。实际 RAG 里经常会把 BM25 和向量检索做混合检索，再进行重排。**

如果压成一段完整回答，大概是：

```text
BM25 是全文检索里常用的相关性打分算法。
它不是语义模型，而是基于词面匹配来判断文档和查询的相关程度。

它主要看三件事：
第一，查询词在文档里出现了多少次，但词频收益会逐渐饱和；
第二，查询词在整个语料库里有多稀有，越稀有越有区分度；
第三，文档长度是否过长，避免长文档因为字多而天然占优。

在 RAG 里，BM25 常用来补关键词精确召回，尤其适合错误码、接口名、配置项这类精确词；它可以和向量检索一起组成混合检索。
```

这样回答就够了。

既讲了定义，也讲了原理，还讲了工程位置。

## 写在最后

所以，BM25 不神秘。

它就是搜索系统里一套经典的关键词打分规则。

它看词频，但不让重复堆词无限加分。

它看稀有度，因为稀有词更能定位文档。

它看文档长度，因为长文档不能靠字多白占便宜。

它不理解复杂语义。

也不是向量检索的替代品。

但在真实系统里，它一直很有用。

尤其当你要找的是错误码、接口名、配置项、类名、产品术语时，BM25 往往比想象中更稳。

很多时候，搜索质量不是靠一个最炫的模型撑起来的。

而是靠这些朴素、稳定、可解释的基础件，一层一层垫起来的。
