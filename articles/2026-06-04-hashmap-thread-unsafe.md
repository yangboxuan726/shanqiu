大家好，我是「山丘代码铺」。

> 这篇文章不逐行抠 HashMap 源码，也不把红黑树、扰动函数、扩容细节全展开。
>
> 只解决一个问题：**HashMap 为什么线程不安全？**
>
> 这类问题在后端面试里非常高频，尤其适合考察候选人对集合、并发和底层结构的理解。

刚开始学 HashMap 的时候，我其实有点低估这个问题。

我当时想得很简单：

> HashMap 不就是一个 key-value 容器吗？
>
> 一个线程 put，一个线程 get，怎么就不安全了？

后来才发现，“线程不安全”不是一句空话。

它背后说的是：

> **HashMap 的内部结构会被修改，而这些修改不是原子操作，也没有同步保护。**

如果多个线程同时改它，就可能出现很奇怪的问题：

- 数据被覆盖；
- 元素丢失；
- size 不准确；
- 遍历时抛出异常；
- 扩容过程中读到不稳定状态。

所以这道题如果只回答一句：

> 因为 HashMap 不是 synchronized 的。

其实有点太薄了。

面试官通常还会继续问：

> 那到底哪里不安全？
>
> put 的时候会发生什么？
>
> resize 的时候为什么更危险？
>
> 那应该用什么替代？

这篇就先把这些问题讲顺。

---

## 01｜先说结论

先给一个面试里的简短答案：

> **HashMap 线程不安全，是因为它没有同步机制。**
>
> **多个线程同时对它做结构性修改时，内部数组、链表、红黑树、size 等状态可能被并发修改，导致数据覆盖、数据丢失、状态不一致，甚至遍历异常。**

这里有几个关键词：

- 没有同步；
- 多线程同时修改；
- 内部结构会变；
- 修改过程不是一步完成；
- 结果没有正确性保证。

Oracle 的 HashMap 文档里也说得很直接：

> 如果多个线程并发访问 HashMap，并且至少有一个线程做结构性修改，就必须在外部做同步。

所谓结构性修改，可以先理解成：

> **会新增或删除映射关系的操作。**

比如：

```java
map.put("a", 1);
map.remove("a");
```

如果只是把已有 key 的 value 改掉，官方文档里不把它算作结构性修改。

但在真实并发场景里，也不要因此觉得：

> 只更新 value 就一定安全。

因为 HashMap 本身仍然没有帮你解决并发下的可见性、原子性和顺序问题。

简单说：

> **HashMap 设计目标是普通单线程 Map，不是并发容器。**

---

## 02｜HashMap 底层到底在改什么？

要理解线程不安全，得先知道 HashMap 大概长什么样。

在 JDK 8 之后，可以先粗略理解成：

```text
数组 + 链表 + 红黑树
```

数组里的每个位置，可以叫一个桶。

一个 key 放到哪个桶里，通常要经过 hash 计算。

大概像这样：

```text
key -> hash -> 数组下标 -> 放进对应桶里
```

如果两个 key 算出来落在同一个桶里，就会发生 hash 冲突。

冲突少的时候，桶里可能是一条链表。

冲突多到一定程度后，链表可能会转成红黑树。

所以你调用一次：

```java
map.put(key, value);
```

看起来是一行代码。

但 HashMap 内部可能做了很多事：

- 计算 hash；
- 找数组下标；
- 判断桶是不是空；
- 判断 key 是否已经存在；
- 插入新节点或更新旧值；
- 修改 size；
- 判断是否需要扩容；
- 扩容时重新分配节点位置。

这些步骤不是一个不可拆分的整体。

多个线程同时进来，就可能互相打断。

这就是问题的开始。

---

## 03｜两个线程同时 put，会发生什么？

先看一个很简单的场景。

现在 HashMap 某个桶是空的。

线程 A 要 put：

```java
map.put("a", 1);
```

线程 B 也要 put：

```java
map.put("b", 2);
```

假设 `"a"` 和 `"b"` 恰好落在同一个桶里。

正常情况下，这个桶里应该能放下两个节点。

但在并发情况下，可能出现这样的过程：

```text
线程 A：看了一眼，这个桶是空的
线程 B：也看了一眼，这个桶是空的

线程 A：把 a 放进去
线程 B：把 b 放进去
```

最后谁覆盖谁，就不好说了。

结果可能变成：

```text
只剩 a
```

也可能变成：

```text
只剩 b
```

你本来 put 了两个 key，最后只留下一个。

这就是典型的数据丢失。

再比如多个线程同时更新 size。

size++ 看起来像一步。

但它背后大概是：

```text
读取 size
加 1
写回 size
```

两个线程同时读到 size = 10。

线程 A 算出 11，写回去。

线程 B 也算出 11，写回去。

明明新增了两个元素，size 可能只增加了 1。

所以 HashMap 的线程不安全，不是说它“偶尔慢一点”。

而是：

> **它内部状态可能真的被改错。**

## 04｜为什么 resize 更容易出问题？

HashMap 不是一开始就开一个无限大的数组。

它有容量，也有负载因子。

默认负载因子通常是 0.75。

可以先简单理解成：

> 当元素越来越多，HashMap 觉得桶快不够用了，就会扩容。

扩容不是简单地把数组长度变大。

它还要把原来的节点搬到新数组里。

这个过程大概像搬家：

```text
旧数组
  ↓
创建新数组
  ↓
重新计算节点位置
  ↓
把节点迁移过去
```

单线程时，这件事没问题。

但多线程时，就麻烦了。

因为一个线程正在搬家，另一个线程可能也在搬家。

或者一个线程正在搬家，另一个线程在读旧结构。

这时候 HashMap 内部就可能处在一种“半迁移”的状态。

面试里经常会听到一个说法：

> JDK 7 的 HashMap 并发扩容可能导致链表成环。

这个说法对应的是老版本实现里的一个经典问题。

但现在面试时不建议只背这句话。

因为到了 JDK 8 之后，HashMap 的实现已经变了，扩容迁移方式和 JDK 7 不完全一样。

更稳的回答是：

> **并发 resize 的核心问题，不只是某个版本会不会成环，而是 HashMap 在扩容迁移时会修改内部结构，这个过程没有同步保护，多线程同时操作就没有正确性保证。**

这样说更不容易被追问卡住。

## 05｜遍历的时候为什么也会出问题？

除了 put 和 resize，遍历 HashMap 时也要小心。

比如你在遍历：

```java
for (String key : map.keySet()) {
    System.out.println(key);
}
```

另一个线程同时在修改：

```java
map.put("newKey", 100);
```

这时候就可能抛：

```text
ConcurrentModificationException
```

这个异常很多人会理解成：

> 只要并发修改，HashMap 就一定会抛异常。

其实不是。

HashMap 的迭代器是 fail-fast 的。

可以理解成：

> 它会尽量快速发现“遍历期间结构被改了”，然后抛异常。

但这个机制不是并发安全保证。

官方文档里也提醒过，fail-fast 只能尽力而为，不能依赖它来保证程序正确性。

也就是说：

> **抛异常，只是帮你发现 bug。**
>
> **不抛异常，也不代表并发修改就是安全的。**

这个点面试里很容易加分。

因为它说明你没有把 `ConcurrentModificationException` 当成并发控制手段。

## 06｜那 Collections.synchronizedMap 可以吗？

可以，但要知道它解决问题的方式。

如果你写：

```java
Map<String, Integer> map =
    Collections.synchronizedMap(new HashMap<>());
```

它会给 HashMap 包一层同步包装。

大概可以理解成：

> 每次访问 map 时，先拿锁，再操作。

这样能避免多个线程同时乱改内部结构。

但它也有代价：

- 并发性能一般；
- 很多操作会被同一把锁串行化；
- 遍历时通常还需要你自己在外面加同步。

比如：

```java
synchronized (map) {
    for (String key : map.keySet()) {
        System.out.println(key);
    }
}
```

所以它能用，但不是高并发场景下最常见的选择。

如果你真的需要一个并发 Map，更常见的是：

```java
ConcurrentHashMap
```

---

## 07｜为什么更推荐 ConcurrentHashMap？

ConcurrentHashMap 不是简单地在 HashMap 外面套一把大锁。

它本来就是为并发场景设计的。

在 JDK 8 里，它会用 CAS、synchronized、volatile 等机制，尽量让多个线程可以更安全地并发访问。

你不用把这些细节一次背完。

面试里先说清楚这个层次就够了：

> **HashMap 是普通 Map，没有内置并发控制。**
>
> **Collections.synchronizedMap 是给普通 Map 包一层同步。**
>
> **ConcurrentHashMap 是专门为并发访问设计的 Map。**

如果只是单线程，或者局部变量里临时用一下：

```java
HashMap
```

完全没问题。

如果多个线程都要访问，并且会修改：

```java
ConcurrentHashMap
```

通常更合适。

## 08｜面试时可以怎么回答？

如果面试官问：

> HashMap 为什么线程不安全？

可以这样回答：

```text
HashMap 本身没有同步机制。

它的 put、remove、resize 等操作都会修改内部结构，比如数组、链表、红黑树和 size。

这些修改不是原子操作，也没有加锁或者其他并发控制。

如果多个线程同时进行结构性修改，可能出现数据覆盖、元素丢失、size 不准确、扩容时状态不一致等问题。

另外，HashMap 的迭代器是 fail-fast 的，遍历期间如果发生结构性修改，可能抛 ConcurrentModificationException。
但这个异常只是尽力检测并发修改，不能当作线程安全保证。

所以多线程修改场景下，不应该直接用 HashMap，可以考虑 Collections.synchronizedMap 或 ConcurrentHashMap。
高并发场景下一般更推荐 ConcurrentHashMap。
```

这个回答比一句“因为 HashMap 没加锁”更完整。

它讲到了：

- 没有同步机制；
- 内部结构会变；
- 并发 put 可能丢数据；
- resize 更危险；
- fail-fast 不是安全保证；
- 替代方案是 synchronizedMap 或 ConcurrentHashMap。

如果面试官继续追问，再展开其中一块就行。

但它其实能牵出很多并发里的核心概念：

- 原子性；
- 可见性；
- 结构性修改；
- fail-fast；
- 扩容；
- 并发容器。

所以它不是让你背一句“HashMap 不是 synchronized 的”。
